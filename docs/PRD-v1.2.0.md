## 一、先说结论（给赶时间的人看）

v1.2.0 要做三件事，按重要性排序：

1. **【你的核心诉求】让 agent 卡片的"头部（head）"可配置、默认关闭** —— 你以为这个功能不存在，其实代码里已经写了一半，但没写进文档、没测试、还有两个漏洞。v1.2.0 要把它补成一个"真正能用"的功能。
2. **【生产日志发现的问题】长对话末尾的日志噪音** —— 生产环境 2.5 天跑了 20 个会话，0 报错，但有一个 12 分钟的长对话在结尾产生了一堆重复日志。这个要优化。实时流式保活作为可选优化，先调研。
3. **【代码体检发现的清理项】删死代码、修文档错误** —— 代码里有一些"写了但从来没人用"的死代码，还有一处 CHANGELOG 和实际代码对不上的错误描述。这些要清理。

**核心原则：v1.1.3 的主流程已经在生产环境验证过（0 报错、100% 流式成功），v1.2.0 不动主流程，只做"补全 + 减噪 + 清理"。**

## 三、需求一：agent 卡片头部（head）可配置【你的核心诉求】

### 3.1 你说的问题，和实际的情况

你说："当前插件 agent 的卡片没有 head，默认关闭的。我期望 head 支持配置，用户可以自由选择开启或者关闭，默认关闭。"

**实际情况**（我们逐行查了代码）：
- 配置项 `header.enabled` **已经存在**于代码里（`config/reader.py` 第 237 行），默认就是关闭（`False`）。
- 创建卡片的代码**已经会读取这个配置**并在开启时加上头部（`controller/linear_mixin.py` 第 225、1708 行）。
- 头部的样式也写好了（`cardkit/elements.py` 第 68 行 `_build_header`）：流式中蓝色"处理中"、完成绿色"已完成"、出错红色"出错"、中断红色"已停止"。

**那你为什么"以为没有"？** 因为这个功能是个**半成品**，有 4 个窟窿：

| 窟窿 | 说明 | 严重程度 |
|------|------|----------|
| ① 完全没文档 | README、中文 README、SKILL.md、AGENT_GUIDE.md 一个字都没提 `header.enabled`；插件首次安装时往配置文件里写的"默认配置"也没有 `header` 这一项。用户根本不知道有这个开关。 | 高 |
| ② 完全没测试 | `tests/` 里没有任何测试验证"开启 header 后卡片真的有头部"。唯一的 相关代码是 e2e 测试框架里强制把它设成 `False`。 | 高 |
| ③ 降级路径没覆盖 | 如果飞书 CardKit 创建卡片失败，插件会降级用旧的 IM 卡片通道。这个降级通道的卡片**不支持 header**。也就是说：用户开了 header，一旦降级，头部会突然消失。 | 中 |
| ④【最关键】封卡后头部颜色不会变 | 这是飞书 API 的限制（见下一节 3.2），不是代码 bug，但它直接决定了我们怎么做这个功能。 | 高 |

### 3.2 一个必须先搞清楚的技术限制（已向飞书官方文档求证）

**问题**：卡片在"流式输出中"时，头部是蓝色"处理中"。等 AI 回复完了，我们想把头部变成绿色"已完成"——飞书 API 允许我们这样做吗？

**求证过程**：我们查了飞书开放平台 CardKit v1 的官方文档，关键发现：

| 飞书 API 接口 | 能不能更新头部？ | 依据 |
|---------------|------------------|------|
| 更新卡片配置（settings，就是关流式、改摘要用的那个） | **不能**。官方原文："支持卡片配置 config 字段和卡片跳转链接 card_link 字段"——只支持 `config` 和 `card_link`，**不包括 header**。 | open.feishu.cn/document/cardkit-v1/card/settings |
| 批量更新（batch_update，流式中增量改内容用的） | **不能**。它的动作只有 add_elements / partial_update_element / delete_element / replace_element，都是改"卡片正文里的元素"，没有改"卡片头部"的动作。 | 官方流式更新概览页 |
| 全量更新（card/update） | **能**。但它会**覆盖整张卡片**，相当于把卡片拆了重建。 | open.feishu.cn/document/cardkit-v1/card/update |

**结论**：在"流式输出中"和"增量封卡"的过程中，**没有办法单独把头部从蓝色改成绿色**。唯一能改头部的方法是"全量更新"——把整张卡片重新发一遍。

**这对我们的影响**：当前代码在创建卡片时设了蓝色"处理中"头部，然后整个流式过程和增量封卡过程都不碰头部。所以如果用户开了 header：
- 流式中：头部显示蓝色"处理中" ✓（正确）
- 封卡后：头部**还是蓝色"处理中"** ✗（应该是绿色"已完成"，但改不了）
- 只有在"封卡失败、走全量重建兜底"这种罕见情况下，头部才会变成正确的完成/错误色

这就是窟窿 ④，v1.2.0 用"方案 B"解决（见 3.3 的 H6）。

### 3.3 v1.2.0 要做的具体事情（决策已固化）

| 编号 | 要做的事 | 通俗解释 | 决策结果 | 影响面 / 风险 |
|------|---------|----------|----------|---------------|
| H1 | 把 `header.enabled` 写进所有文档 | 让用户知道有这个开关、怎么开 | ✅ 做 | **零风险**，纯加文档 |
| H2 | 把 `header: {enabled: false}` 加进插件首次安装时写入的默认配置 | 用户装完插件打开配置文件就能看到这一项 | ✅ 做 | **极低风险**。只对"新安装"生效，不覆盖已有配置 |
| H3 | `/aowen status` 命令的配置面板里展示 header 开关状态 | 用户在飞书里发 `/aowen status` 就能看到 header 开了没 | ✅ 做 | **极低风险** |
| H4 | 启动时的诊断日志里加上 header 状态 | 排障时能看到 header 配置 | ✅ 做 | **零风险** |
| H5 | 补测试 | 给 header 功能写测试 | ✅ 做 | **零风险**，纯加测试 |
| H6 | 解决"封卡后头部颜色不变"问题 | **决策 1 = 方案 B**：开了 header 就改用"全量重建"封卡，让头部颜色正确（蓝→绿/红） | ✅ 方案 B | **中等风险**，会动到封卡流程入口分支。详见 DESIGN 第二章 |

## 六、不做什么（明确排除项）

为了避免 v1.2.0 范围蔓延，以下事项**明确不做**：

1. **不改 CardKit 主流程**（创建卡片 → 流式更新 → 增量封卡）。这套流程在生产中 100% 成功，动了风险太大。
2. **不改降级方案**（drain fallback、IM 降级）。生产中只触发 1 次且成功保住内容，保留作保险。
3. **不做并发限流的改动**。v1.1.0 加的并发限流在生产中 0 次触发，但留着当保险，不动。
4. **不重构 monkey patching 体系**。7 次重启全部打补丁成功，不动。
5. **不做 header 的"自定义标题文案/自定义图标"**。v1.2.0 只做"开关 + 状态色"，自定义内容留到未来版本。
6. **不补 mtime 自动检测**（决策 4）。这是 v1.1.0 已明确删除的设计，v1.2.0 只纠正文档描述。
